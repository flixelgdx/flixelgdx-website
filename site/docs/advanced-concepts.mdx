# Advanced Concepts

This page covers four systems that are optional but unlock a significant step up in code quality and
flexibility once you are comfortable with the core framework.

---

## Action sets — platform-agnostic input

Action sets decouple your game logic from specific hardware. Instead of asking "is the Space key down?"
you ask "is the Jump action pressed?" — and the framework figures out whether that came from a keyboard,
a controller, or a touch screen.

### Defining an action set

Extend `FlixelActionSet` and declare each action as a field. In the constructor, add bindings. Key and
gamepad codes are plain `int` constants on `FlixelKey` and `FlixelGamepadInput`:

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
public class PlayerControls extends FlixelActionSet {

  public final FlixelActionDigital jump;
  public final FlixelActionDigital attack;
  public final FlixelActionAnalog move;

  public PlayerControls() {
    // add(...) is the protected helper inherited from FlixelActionSet, so we
    // call it inside the constructor to add the binds to our actions.
    jump = new FlixelActionDigital("jump");
    jump.addBinding(FlixelInputBinding.key(FlixelKey.SPACE));
    jump.addBinding(FlixelInputBinding.key(FlixelKey.W));
    jump.addBinding(FlixelInputBinding.gamepadButton(0, FlixelGamepadInput.A));
    add(jump);

    attack = new FlixelActionDigital("attack");
    attack.addBinding(FlixelInputBinding.key(FlixelKey.Z));
    attack.addBinding(FlixelInputBinding.gamepadButton(0, FlixelGamepadInput.X));
    add(attack);

    move = new FlixelActionAnalog("move");
    move.addAxisBinding(FlixelAnalogAxisBinding.negXKey(FlixelKey.LEFT));
    move.addAxisBinding(FlixelAnalogAxisBinding.posXKey(FlixelKey.RIGHT));
    move.addAxisBinding(FlixelAnalogAxisBinding.negYKey(FlixelKey.UP));
    move.addAxisBinding(FlixelAnalogAxisBinding.posYKey(FlixelKey.DOWN));
    move.addAxisBinding(FlixelAnalogAxisBinding.gamepadAxisX(0, FlixelGamepadInput.AXIS_LEFT_X));
    move.addAxisBinding(FlixelAnalogAxisBinding.gamepadAxisY(0, FlixelGamepadInput.AXIS_LEFT_Y));
    add(move);
  }
}
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin

  }

  }
}
```

### Using the action set

Instantiate it once (typically in your state's `create`) and poll it in `update`. Digital actions expose
`pressed()`, `justPressed()`, and `justReleased()`; analog actions expose `getX()` and `getY()`:

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
private final PlayerControls controls = new PlayerControls();

@Override
public void update(float elapsed) {
  super.update(elapsed);
  if (controls.jump.justPressed()) {
    player.setVelocityY(-JUMP_SPEED);
  }
  if (controls.attack.pressed()) {
    attackTimer -= elapsed;
  }
  player.setVelocityX(controls.move.getX() * SPEED);
}
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
private val controls = PlayerControls()

override fun update(elapsed: Float) {
  super.update(elapsed)
  if (controls.jump.justPressed()) player.velocityY = -JUMP_SPEED
  if (controls.attack.pressed()) attackTimer -= elapsed
  player.velocityX = controls.move.x * SPEED
}
```

</TabItem>
</Tabs>

`FlixelActionSet` is registered with `FlixelActionSets` automatically, so the framework calls `update`
and `endFrame` on it every game loop tick. `destroy()` the set when you no longer need it (e.g., on
state transition) to unregister it and prevent stale input reads.

## Adobe Animate integration — FlixelAnimateSprite

`FlixelAnimateSprite` plays animations exported from **Adobe Animate** as texture atlas sets (the
`spritemap1.png` / `spritemap1.json` / `Animation.json` triple that Animate's exporter produces).
It is a drop-in replacement for `FlixelSprite` with a full nested-symbol timeline engine.

### Loading an animation

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
FlixelAnimateSprite character = new FlixelAnimateSprite(x, y);
character.addSpritemapAndAnimation(
  "path/to/spritemap1.png",   // texture atlas image
  "path/to/spritemap1.json",  // atlas layout
  "path/to/Animation.json"    // clip/keyframe data
);
character.animation.playAnimation("Walk");
add(character);
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
val character = FlixelAnimateSprite(x, y).apply {
  addSpritemapAndAnimation(
    "path/to/spritemap1.png",  // texture atlas image
    "path/to/spritemap1.json", // atlas layout
    "path/to/Animation.json"   // clip/keyframe data
  )
  animation?.playAnimation("Walk")
}
add(character)
```

</TabItem>
</Tabs>

### Multiple sheets and symbols

If the animation spans multiple sprite sheets (common for large characters or merged exports), call
`addSpritemapAndAnimation` again with the next triple. Later calls merge their clips into the same
controller, so all animation names become available on the same `playAnimation` API.

### Controlling playback

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
FlixelAnimationController anim = character.animation;
anim.playAnimation("Run");            // start a clip by name (loops by default)
anim.playAnimation("Attack", false);  // play once, no looping
anim.pause();                         // pause on the current frame
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
val anim = character.animation
anim?.playAnimation("Run")            // start a clip by name (loops by default)
anim?.playAnimation("Attack", false)  // play once, no looping
anim?.pause()                         // pause on the current frame
```

</TabItem>
</Tabs>

`FlixelAnimateSprite` supports all standard `FlixelSprite` properties — position, velocity, physics,
flipping, scaling, and color tint — so you can treat it identically to any other sprite.

## Advanced asset management

`Flixel.assets` is a `FlixelAssetManager` that wraps libGDX's `AssetManager` as a loading engine and
keeps a single unified cache of `FlixelAsset<T>` handles at runtime. Every asset type - images, audio,
text - is retrieved through the same interface. It is the right tool for games that need to control when
assets are loaded and released rather than keeping everything in memory for the lifetime of the game.

### The FlixelAsset handle

Every asset the manager returns is a `FlixelAsset<T>`, where `T` is the wrapper type game code actually
uses. Images return `FlixelAsset<FlixelGraphic>`; audio sources return `FlixelAsset<FlixelSoundSource>`;
text files return `FlixelAsset<String>`. Call `handle.get()` to access the content.

`FlixelGraphic` implements `FlixelAsset<FlixelGraphic>` directly, so `handle.get()` returns the graphic
itself and there is no extra unwrapping step. Use `graphic.getTexture()` only when you need the raw
libGDX `Texture` (for example, to split a spritesheet by hand).

### Loading assets

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
// Queue assets for async loading. The file extension selects the loader automatically.
Flixel.assets.load("textures/tileset.png");
Flixel.assets.load("audio/music/stage1.ogg");

// Mark an asset as persistent so it survives state switches even at zero references.
Flixel.assets.load("textures/ui/hud.png", true);

// Block until all queued loads complete (use only inside a loading state).
while (!Flixel.assets.update()) {
  progressBar.setValue(Flixel.assets.getProgress());
}
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">


### Retrieving and releasing assets

`get(path)` returns a `FlixelAsset<T>` handle and loads synchronously if the asset was not queued
ahead of time. Call `retain()` when you take ownership and `release()` when you are done - the manager
uses the reference count to decide when it is safe to unload the underlying data.

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
// Retrieve a handle and take ownership. If not already loaded, this blocks.
// Prefer loading in a loading state first to avoid mid-frame stalls.
FlixelAsset<FlixelGraphic> handle = Flixel.assets.get("textures/tileset.png");
handle.retain();
FlixelGraphic graphic = handle.get();

// Access the raw texture if you need it (for manual splitting, etc.).
Texture tex = graphic.getTexture();

// Release in destroy() to let the manager unload it at the next state switch.
handle.release();
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
// Retrieve a handle and take ownership. If not already loaded, this blocks.
// Prefer loading in a loading state first to avoid mid-frame stalls.
val handle = Flixel.assets.get<FlixelGraphic>("textures/tileset.png")
handle.retain()
val graphic: FlixelGraphic = handle.get()

// Access the raw texture if you need it (for manual splitting, etc.).
val tex: Texture = graphic.getTexture()

// Release in destroy() to let the manager unload it at the next state switch.
handle.release()
```

</TabItem>
</Tabs>

Use `peek(path)` for a non-owning read that does not change the reference count - useful for checking
whether a shared resource is already registered before creating it.

### Loading state pattern

For large games, use a dedicated loading state that shows a progress bar:

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
public class LoadingState extends FlixelState {

  private FlixelBar progressBar;

  @Override
  public void create() {
    super.create();

    Flixel.assets.load("textures/level1.png");
    Flixel.assets.load("audio/music/level1.ogg");
    // ... more queued loads

    float barW = Flixel.getViewWidth() * 0.6f;
    float barH = 20f;
    progressBar = new FlixelBar(
        (Flixel.getViewWidth() - barW) / 2f,
        (Flixel.getViewHeight() - barH) / 2f,
        barW,
        barH);
    progressBar.setTrack(Flixel.assets::getProgress);
    add(progressBar);
  }

  @Override
  public void update(float elapsed) {
    super.update(elapsed);
    if (Flixel.assets.update()) {
      Flixel.switchState(new PlayState());
    }
  }
}
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin

  private lateinit var progressBar: FlixelBar

  override fun create() {

}
```

### Custom asset types

Register a loader for any file extension you need. The `FlixelAssetLoader` functional interface takes
the manager and the path and returns a fully constructed `FlixelAsset<T>` handle.

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
// Load .cfg files as plain text strings.
Flixel.assets.registerLoader(".cfg", String.class,
    (assets, path) -> new FlixelDefaultAsset<>(assets, path, String.class));

// Now load and retrieve exactly like any built-in type.
Flixel.assets.load("data/settings.cfg");
Flixel.assets.finishLoading();
FlixelAsset<String> cfg = Flixel.assets.get("data/settings.cfg");
String text = cfg.get();
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
// Load .cfg files as plain text strings.
Flixel.assets.registerLoader(".cfg", String::class.java) { assets, path ->
  FlixelDefaultAsset(assets, path, String::class.java)
}

// Now load and retrieve exactly like any built-in type.
Flixel.assets.load("data/settings.cfg")
Flixel.assets.finishLoading()
val cfg = Flixel.assets.get<String>("data/settings.cfg")
val text: String = cfg.get()
```

</TabItem>
</Tabs>

### Asset modes

`FlixelAssetMode` controls when unreferenced, non-persistent assets are reclaimed. Set it once in your
game's setup:

| Mode | When assets are freed |
|------|----------------------|
| `LAZY` | Never automatically - you call `clearNonPersist()` or `clear()` yourself. |
| `STANDARD` | On every `Flixel.switchState(...)` call (the default). |
| `AGGRESSIVE` | Immediately when an asset's reference count drops to zero. |

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
Flixel.assets.setAssetMode(FlixelAssetMode.AGGRESSIVE);
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
Flixel.assets.assetMode = FlixelAssetMode.AGGRESSIVE
```

</TabItem>
</Tabs>

`AGGRESSIVE` is best for games with many short-lived assets where you want immediate GPU memory
reclaim. `STANDARD` is the safe default for most games. `LAZY` gives you full manual control.

## The logging API

`FlixelLogger` is the framework's structured logging system. You write to it through the static helpers
on `Flixel`, and it delivers each entry to both the console and (optionally) a rolling log file. The
debug overlay's Log window listens to it in real time.

### Writing log messages

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
// Three severity levels, each with an optional tag.
Flixel.info("Spawner", "Spawned enemy at (100, 200)");
Flixel.warn("Physics", "Velocity clamped: " + obj.getVelocityX());
Flixel.error("SaveSystem", "Failed to write save file", exception);
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
// Three severity levels, each with an optional tag.
Flixel.info("Spawner", "Spawned enemy at (100, 200)")
Flixel.warn("Physics", "Velocity clamped: ${obj.velocityX}")
Flixel.error("SaveSystem", "Failed to write save file", exception)
```

</TabItem>
</Tabs>

The tag is free-form text. Use it as a module/subsystem identifier so you can filter messages in the
Log window (or `grep` in a log file) when debugging a specific system.

### Enabling file logging

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
// In your game's constructor or create():
Flixel.log.startFileLogging();
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">


### Log file location

| Context | Default location |
|---------|-----------------|
| Running from IDE | `<project root>/logs/` |
| Running from a JAR | `logs/` next to the JAR |
| Custom | Set via `Flixel.log.setLogsFolder("path/to/logs")` |

### Custom log listeners

Attach a listener to receive every log entry as a `FlixelLogEntry` (a record exposing `level()`, `tag()`,
and `message()`):

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
Flixel.log.addLogListener(entry -> {
  if (entry.level() == FlixelLogLevel.ERROR) {
    crashReporter.submit(entry.tag(), entry.message());
  }
});
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">

```kotlin
Flixel.log.addLogListener { entry ->
  if (entry.level() == FlixelLogLevel.ERROR) {
    crashReporter.submit(entry.tag(), entry.message())
  }
}
```

</TabItem>
</Tabs>

Listeners receive entries even in `RELEASE` builds where the debug overlay is absent. This makes them
the right hook for crash reporting, telemetry, or an in-game dev console of your own design.

### Log modes

| Mode | What it prints |
|------|---------------|
| `SIMPLE` | File location + line + message only. |
| `DETAILED` | Level + timestamp + tag + message. Always used for file output. |

<Tabs groupId="lang">
<TabItem value="java" label="Java" default>

```java
Flixel.log.setLogMode(FlixelLogMode.DETAILED);
```

</TabItem>
<TabItem value="kotlin" label="Kotlin">
